## Rule #1 🔴 CRITICAL — Workflow при получении задачи
**Created:** 2026-04-17
**Status:** active
**Rule:** При получении задачи от владельца — ВСЕГДА выполнять следующий алгоритм:
1. Уточнить с каким проектом работаем
2. Обратиться к PROJECTS_TASKS.md (VPS DO: `/opt/openclaw_vps/workspace/PROJECTS_TASKS.md`, SSH: `sshpass -p 'HouseTeka=110110-Semen' ssh root@45.55.225.86`)
3. Проверить — существует ли проект в PROJECTS_TASKS.md
4. Если ДА — спросить: "Работаем с проектом [название]?" — ждать подтверждения
5. Если НЕТ → спросить: "Создать проект [название]?" — после подтверждения:
   a. Создать проект в PROJECTS_TASKS.md с обязательными полями:
      - **GitHub:** ссылка на репозиторий
      - **Бэкапы:** сервер + путь (Rule #7)
   b. Создать GitHub repo + начальный коммит
   c. Создать TZ.md из шаблона (Rule #5)
   d. Создать папку бэкапов на сервере (Rule #7)
   e. Создать PROJECT_CHANGELOG.md
6. Прочитать ВСЕ задачи проекта: какие реализованы, какие в работе, последний коммит в GitHub
7. Если задача продолжение существующей — приступить к выполнению
8. Если задача НОВАЯ: записать задачу → сделать бэкап → выполнить
9. Когда задача ВЫПОЛНЕНА: коммит в GitHub → статус задачи → записать в память
10. ВСЁ логировать в RULES_LOG.md
**Scope:** global

## Rule #2 🔴 CRITICAL — PROJECT_CHANGELOG (append-only)
**Created:** 2026-04-17
**Status:** active
**Rule:** Каждый проект имеет `PROJECT_CHANGELOG.md` в папке проекта. Формат записи:
```
## [YYYY-MM-DD HH:MM UTC] — [Краткое описание]
**Задача:** что делали
**Что сделано:** конкретные изменения (файлы, функции, компоненты)
**Коммит:** хеш
**Ответственный:** кто делал
**Статус:** ✅ Работает / ⚠️ Частично / ❌ Не работает
**Проверил владелец:** ДА / НЕТ / Ждёт проверки
```
КАЖДОЕ действие по проекту записывается. Append-only: ❌ нельзя удалять/редактировать старые записи. ✅ только добавлять.
**Scope:** global

## Rule #4 🔴 CRITICAL — Диагностика "почему не работает?"
**Created:** 2026-04-17

**Scope:** global

## Rule #5 🔴 CRITICAL — Техническое Задание (TZ.md) под каждый проект
**Created:** 2026-04-17
**Status:** active
**Rule:** Под каждый проект создаётся файл `TZ.md` (Техническое Задание).

### Где хранится:
- VPS DO: `/opt/openclaw_vps/workspace/projects/[название-проекта]/TZ.md`
- Доступен через веб-версию менеджера проектов
- Доступен через ИИ агента (голос/текст → агент обновляет файл)

### Что содержит TZ.md:
- Общая информация (цель, статус, стек, команда)
- Функциональные требования (фичи с чекбоксами и статусами)
- Технический стек
- Ограничения и правила

### Статусы фич: ⬜ Не начато / 🔄 В работе / ✅ Готово / ❌ Не работает

### Связь с PROJECTS_TASKS.md:
- Задачи ставятся НА ОСНОВАНИИ TZ.md
- При завершении задачи → обновить статус фичи в TZ.md

### Append-only для истории:
- Фича меняется → старое не удалять, добавлять `📌 Обновлено: [дата]`
**Scope:** global

## Rule #6 🔴 CRITICAL — Сохранение и память
**Created:** 2026-04-17
**Status:** active
**Rule:** Три уровня сохранения по команде владельца.

### "Сохрани" / "Сохранись":
1. Записать в дневник сессии: `memory/YYYY-MM-DD.md` (что делали, какие решения, промежуточные результаты)
2. Обновить PROJECTS_TASKS.md (статусы задач, что сделано с последнего сохранения)
3. Обновить PROJECT_CHANGELOG.md если были изменения по проекту
4. Подтвердить: "✅ Сохранил: дневник + менеджер проектов"

### "Сохрани в глубокую память":
1. Выполнить всё из "Сохрани"
2. Обновить MEMORY.md (важные решения, ключевые изменения, уроки)
3. Спросить: "Пушить в Git?" — ждать ответа
   - Если ДА: проверить есть ли репозиторий → есть: git add + commit + push. Нет: создать репо на GitHub + push
   - Если НЕТ: пропустить
4. Подтвердить: "✅ Глубокая память + Git обновлён"

### "Пуш в git":
1. Определить текущий проект и его репозиторий
2. git add -A → git commit с описанием изменений → git push
3. Если репозитория нет → создать на GitHub + push
4. Подтвердить с хешем коммита

### Файл SERVERS.md (Серверы и доступы):
- **Расположение:** Workspace корень каждого агента
- **Формат:** таблица (имя агента, сервер, IP, SSH пароль, Telegram бот, назначение)
- **Кто меняет:** только владелец (или агент по явной команде)

### "Сохрани доступ" / "Сохрани сервер":
1. Спросить: IP, SSH пароль, имя агента, Telegram бот (если есть)
2. Записать в SERVERS.md
3. Подтвердить

### "Установи на сервер X такой-то скилл" / "Отправь файл на сервер X":
1. Открыть SERVERS.md → найти сервер X
2. Если ЕСТЬ → использовать SSH доступы для установки/отправки
3. Если НЕТ → "У меня нет доступов к серверу X. Пришли IP и SSH пароль, я сохраню."
**Scope:** global

## Rule #7 🔴 CRITICAL — Бэкапы при создании проекта
**Created:** 2026-04-17
**Status:** active
**Rule:** При создании нового проекта — сразу создать директорию для бэкапов.

### Алгоритм:
1. Спросить владельца: "На каком сервере хранить бэкапы проекта [название]?"
2. Владелец отвечает (например: "на Максе" или IP или имя агента)
3. Определить сервер из SERVERS.md или AGENT_NETWORK.md
4. Создать директорию: `/backups/[название-проекта]/`
5. Создать файл `REPO_INFO.md` в папке бэкапов:
```
# [Название проекта] — Backup Info
- **GitHub:** ссылка на репозиторий
- **Сервер бэкапов:** IP / имя агента
- **Путь:** /backups/[название-проекта]/
- **Максимум копий:** 10
- **Создан:** YYYY-MM-DD
```
6. Уведомить ИИ агента на сервере бэкапов:
   - Через SSH отправить сообщение в его RULES_LOG.md:
   - `[ДАТА] ВНИМАНИЕ: Создана папка бэкапов для проекта [название] на этом сервере (/backups/[название]/). Хранить до 10 последних копий.`
7. Записать путь к бэкапам в PROJECTS_TASKS.md (секция проекта):
   `**Бэкапы:** [агент/сервер] /backups/[название]/ (макс 10 копий)`
8. Подтвердить владельцу: "✅ Папка бэкапов создана на [сервер], агент [имя] уведомлён."

### При бэкапе проекта:
- Делать бэкап в папку на сервере бэкапов (если доступен через SSH)
- Хранить максимум 10 последних копий (удалять старые)
- Имя файла: `[название]-YYYY-MM-DD_HHMM.tar.gz`
**Scope:** global
